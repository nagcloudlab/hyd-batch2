Security Audit Findings

A real-world security audit on an unsecured calendar app (JBCP)

Unrestricted Admin Access

Any user can access /admin directly. No login required.

No Role-Based Access

All users have same privileges. No differentiation between regular users and admins.

No Password Encryption

Passwords stored in plain text. One DB breach = all accounts compromised.

No CSRF Protection

Forms can be submitted from malicious sites. Cross-site request forgery wide open.

No Ownership Validation

Users can view/edit other users' data. No data isolation.

No Audit Trail

No logging of who accessed what, when, or from where.

Why Security

Architecture & Security

Your architecture directly impacts where and how security is applied

Monolith

Single unit deployment. If one part is compromised, entire app is exposed. Coarse-grained security.

N-Tier / MVC

Separated layers. Security checkpoints at each layer: UI validation, service access control, data filtering.

SOA

Independent services. Must authenticate each call and secure service endpoints. Role consistency is hard.

Microservices

Self-contained services. Needs API gateway, mTLS, identity propagation, JWT/OAuth2. Each service individually protected.

As architecture becomes more distributed, security must become decentralized, granular, and automated.

Core Architecture

Architecture

Servlet Filters Recap

Spring Security is built on top of the Servlet Filter API

HTTP Request
↓
Filter 1 (Logging)
↓
Filter 2 (Security)
↓
Servlet (DispatcherServlet)
Servlet Filters intercept requests before they reach your controller. Spring Security adds its own filters to this chain.
Architecture

DelegatingFilterProxy

Bridge between Servlet container and Spring context

Servlet Container (Tomcat)
↓ knows about Servlet Filters
DelegatingFilterProxyStandard Servlet Filter - delegates to Spring bean
↓ delegates to
FilterChainProxySpring bean: "springSecurityFilterChain"
Why? Servlet container doesn't know about Spring beans. DelegatingFilterProxy bridges the gap. Spring Boot auto-configures this.
Architecture

FilterChainProxy

Can hold multiple SecurityFilterChains - first match wins

FilterChainProxy
↓ selects matching chain
SecurityFilterChain #1/api/** → JWT filters (stateless)
SecurityFilterChain #2/admin/** → Form login filters
SecurityFilterChain #3/** → Default filters
Order matters! First chain that matches the URL pattern handles the request. Use @Order to control priority.
Architecture

SecurityFilterChain

The @Bean you configure - defines your security rules

Java
@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/").permitAll()
            .anyRequest().authenticated()
        )
        .formLogin(Customizer.withDefaults());
    return http.build();  // builds a SecurityFilterChain
}
HttpSecurity is the builder DSL. It assembles the ordered list of Security Filters that form your SecurityFilterChain.

Architecture

Adding Custom Filters

Java
// Add BEFORE a specific filter (most common: JWT filter)
http.addFilterBefore(myJwtFilter, UsernamePasswordAuthenticationFilter.class);

// Add AFTER a specific filter
http.addFilterAfter(myLoggingFilter, AuthorizationFilter.class);

// Add AT the same position (replaces)
http.addFilterAt(myCustomAuthFilter, UsernamePasswordAuthenticationFilter.class);
Most common: Adding a JWT filter before UsernamePasswordAuthenticationFilter so JWT tokens are validated before Spring tries form login.

AuthenticationManager

One interface, one method

Interface
public interface AuthenticationManager {

        throws AuthenticationException;
}
Takes an unauthenticated token (with credentials). Returns a fully authenticated one (with authorities). Or throws BadCredentialsException.

ProviderManager

Default implementation - iterates through providers

ProviderManagerHas List<AuthenticationProvider>
↓ tries each provider
Provider #1: supports()? → No → skip
↓
Provider #2: supports()? → Yes → authenticate!
↓
Provider #3: (never reached)
First match wins. If none supports the token type, throws ProviderNotFoundException.

UserDetails Interface

Interface
public interface UserDetails {
    String getUsername();
    String getPassword();                     // hashed!
    Collection<? extends GrantedAuthority> getAuthorities();
    boolean isAccountNonExpired();
    boolean isAccountNonLocked();
    boolean isCredentialsNonExpired();

}
Four boolean flags for fine-grained account control. All must be true for login to succeed.
Authentication

PasswordEncoder

BCryptPasswordEncoder

Default. Adaptive hash + random salt. Cost factor 10. Intentionally slow.

Argon2PasswordEncoder

Modern. Memory-hard. Resistant to GPU attacks.

SCryptPasswordEncoder

CPU + memory hard. Good alternative.

NoOpPasswordEncoder

NEVER in production! Plain text. Testing only.

Example
encoder.encode("password")  → "$2a$10$N9qo8uLO..."  // different each time!
encoder.matches("password", "$2a$10$N9qo...")  → true

Authentication

Programmatic Login via SecurityContextHolder

Auto-login after signup - no redirect to login form

Java
// After user signs up, auto-login them:

// 1. Create UserDetails
UserDetails userDetails = new User(
    newUser.getEmail(), newUser.getPassword(),
    AuthorityUtils.createAuthorityList("ROLE_USER"));

// 2. Create Authentication token
Authentication auth = new UsernamePasswordAuthenticationToken(
    userDetails, newUser.getPassword(), userDetails.getAuthorities());

// 3. Set in SecurityContext - user is now "logged in"
SecurityContextHolder.getContext().setAuthentication(auth);
Use cases: After signup, OAuth2/Social login mapping, guest sessions, mobile APIs. Bypasses Spring Security filters.
Authentication

Custom AuthenticationProvider

Full control over validation logic - domain-based, multi-tenant, OTP

Java
@Component
public class CalendarUserAuthenticationProvider implements AuthenticationProvider {

    @Override
    public Authentication authenticate(Authentication auth) {
        String email = auth.getName();
        String password = (String) auth.getCredentials();

        CalendarUser user = calendarService.findUserByEmail(email);
        if (user == null || !passwordEncoder.matches(password, user.getPassword()))
            throw new BadCredentialsException("Invalid credentials");

        var authorities = CalendarUserAuthorityUtils.createAuthorities(user);
        return new UsernamePasswordAuthenticationToken(user, password, authorities);
    }

    @Override
    public boolean supports(Class<?> auth) {
        return UsernamePasswordAuthenticationToken.class.equals(auth);
    }
}

Section 6

Authorization Deep Dive

Authorization

URL-Based Rules

Java
http.authorizeHttpRequests(auth -> auth
    .requestMatchers("/", "/about").permitAll()              // public
    .requestMatchers("/admin/**").hasRole("ADMIN")            // admin only
    .requestMatchers("/events/").hasRole("ADMIN")            // admin only
    .requestMatchers("/**").hasRole("USER")                 // logged in
    .requestMatchers("/resources/**").permitAll()            // static files
);
Order matters! Rules evaluated top to bottom, first match wins. Put specific rules before general ones. End with .anyRequest().
Authorization

hasRole vs hasAuthority

hasRole("ADMIN")

Auto-prepends ROLE_ prefix.
Checks for ROLE_ADMIN.
Convenience for role-based checks.

hasAuthority("ROLE_ADMIN")

Uses exact string.
No prefix added.
Works for any authority, not just roles.

Equivalent
.hasRole("ADMIN")                     // checks ROLE_ADMIN
.hasAuthority("ROLE_ADMIN")            // checks ROLE_ADMIN
.hasAuthority("SCOPE_read")            // OAuth2 scope (only hasAuthority)
Authorization

Method-Level: @PreAuthorize

Defense in depth - secure at method level, not just URLs

Java
@EnableMethodSecurity  // on @Configuration class

@PreAuthorize("hasRole('ADMIN')")
public void deleteUser(Long id) { ... }

@PreAuthorize("hasRole('ADMIN') or #username == authentication.name")
public User getUser(String username) { ... }

@PostAuthorize("returnObject.owner == authentication.name")
public Document getDocument(Long id) { ... }

@PostFilter("filterObject.owner == authentication.name")
public List<Document> getAll() { ... }
Defense in depth: URL rules = first gate. Method security = second gate. Even if URL check is bypassed, method security catches it.

Features

Custom Login Page

Replace the default with your branded login

Java
.formLogin(form -> form
    .loginPage("/login/form")           // URL that shows login form
    .loginProcessingUrl("/login")       // URL that receives POST
    .failureUrl("/login/form?error")     // redirect on failure

Important: Permit access to login page! .requestMatchers("/login/**").permitAll() — otherwise infinite redirect loop.
Features

Session Management

Session Fixation Protection

New session ID after login. Prevents pre-set session attacks. Default: migrateSession().

Concurrent Sessions

maximumSessions(1) - one session per user. New login kicks old one out.

Session Timeout

server.servlet.session.timeout=30m. Idle sessions expire.

Creation Policy

ALWAYS, IF_REQUIRED, NEVER, STATELESS. Use STATELESS for JWT.

CORS

Frontend localhost:3000 → Backend localhost:8080 = Blocked!

Java
http.cors(cors -> cors.configurationSource(request -> {
    var config = new CorsConfiguration();
    config.setAllowedOrigins(List.of("http://localhost:3000"));
    config.setAllowedMethods(List.of("GET","POST","PUT","DELETE"));
    config.setAllowedHeaders(List.of("*"));
    config.setAllowCredentials(true);
    return config;
}));
CORS headers tell the browser: "This origin is allowed." The browser enforces CORS, not the server.

Section 10

Best Practices & Pitfalls

Best Practice

Always Hash Passwords

NEVER

password = "admin123"
MD5("admin123")
SHA1("admin123")

ALWAYS

BCryptPasswordEncoder
Argon2PasswordEncoder
SCryptPasswordEncoder

BCrypt is the default. Intentionally slow, random salt, different output each time.
Best Practice

Default to Deny

BAD - Opt-out

.anyRequest().permitAll()
New endpoints auto-public. Forgotten = exposed.

GOOD - Opt-in

.anyRequest().authenticated()
New endpoints require auth. Explicitly open what's public.

Principle of least privilege. Start locked. Open selectively.
Best Practice

Defense in Depth

Layer 1: URL RulesauthorizeHttpRequests()
↓
Layer 2: Method Security@PreAuthorize on methods
↓
Layer 3: Input Validation@Valid, sanitization
↓
Layer 4: HTTPS + HeadersTLS, security headers
Multiple layers. If one fails, the next catches it. Never rely on a single check.
Pitfalls

Common Mistakes

Plain text passwords

One DB breach = all accounts compromised.

Disabling CSRF carelessly

Only safe for truly stateless APIs.

Hardcoded JWT secrets

In application.properties, committed to Git.

JWT in localStorage

XSS can steal tokens. Use httpOnly cookies.

No token expiry

Permanent access if stolen.

Client-side only checks

Hiding UI != security. Validate server-side.
