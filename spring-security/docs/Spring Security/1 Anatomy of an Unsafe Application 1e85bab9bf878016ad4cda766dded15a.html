1. Anatomy of an Unsafe Application

“Instead of introducing Spring Security first, let’s examine what happens when a web app is built without any security at all.”

🔷 Understanding Architecture Styles and Security Challenges

Application architecture directly impacts how and where security controls should be applied.

✅ 1. Monolithic Architecture

  • The entire application is deployed as a single unit (one WAR or binary).
  • Often used in older or small-scale systems.
  • Drawback: If one part is compromised, the entire app is exposed.
  • Security controls are typically coarse-grained and harder to isolate.

✅ 2. N-Tier / Layered / MVC Architecture

  • The application is split into layers:
    • Presentation (Controller)
    • Service (Business logic)
    • Persistence (DAO, database)
  • Encourages separation of concerns.
  • Allows security to be applied at:
    • UI level → Input validation
    • Service level → Access control
    • DAO level → Data filtering
  • This model is common in Spring MVC applications.

✅ 3. Service-Oriented Architecture (SOA)

  • Application is composed of independent, reusable services.
  • Each service has a well-defined interface.
  • Services communicate over the network via SOAP or REST.
  • Requires authenticating each call and securing service endpoints.
  • Drawback: Role-based permissions often become too generic or inconsistent.

✅ 4. Microservices Architecture

  • Each module is a self-contained service, often running in containers.
  • Promotes agility and scalability.
  • Communicate via REST, RPC, messaging.
  • 🔒 Microservices shift security from the app to infrastructure and API gateways.
  • Introduces new security needs:
    • Identity propagation (carry auth across services)
    • Service-to-service encryption (mTLS)
    • API gateway for entry point and enforcement
    • Token-based authentication (OAuth2, JWT)
  • Each service must be individually protected.

Lesson:

As the architecture becomes more distributed, security must become decentralized, granular, and automated.

🔷 Introducing the Case Study App – JBCP Calendar

You’ll work with a deliberately insecure app called JBCP (Jim Bob Circle Pants ) Calendar to observe and fix common security flaws.

✅ App Design

  • Built with Spring MVC (no Spring Boot).
  • Uses:
    • Controllers for HTTP endpoints
    • Service layer for business logic
    • DAO layer for JDBC + H2 database access
  • UI is made with Template Engine ( Thymeleaf )
  • No authentication, no role checks, and no ownership validation.

Why this app?

  • Provides a realistic, layered Spring web application.
  • Designed to be vulnerable on purpose so you can harden it step-by-step.

🔷 Security Audit: A Realistic Scenario

Imagine a security team conducts an audit on the live JBCP Calendar app.

Their findings are severe.

✅ Security Audit Findings

  1. Unrestricted Admin Access
    • Any user can access /admin directly — no login required.
  1. No Role-Based Access
    • All users have the same privileges.
    • No differentiation between regular users and administrators.
  1. Passwords Stored in Plaintext
    • Credentials are saved without encryption or hashing.
    • Anyone with DB access can see real passwords.
  1. Unencrypted Communication
    • The app uses plain HTTP.
    • Login credentials can be sniffed by network attackers.
  1. Sensitive Information Leaked
    • User data like name, email, and event details are publicly exposed.
    • No control over who accesses what.
  1. Missing CSRF and XSS Protection
    • Forms are open to cross-site request forgery attacks.
    • User inputs are not escaped — vulnerable to JavaScript injection.
  1. SQL Injection Risk
    • Inputs are not validated or sanitized before database use.
    • Application is open to malicious SQL statements.

📌 Because of these issues, the application must be taken offline until all problems are resolved.

🔷 Legal and Compliance Responsibilities

Security isn’t just best practice — it’s legally required in many industries.

✅ Key Regulatory Areas

  • PCI-DSS – For apps handling payment card information.
  • HIPAA – For apps processing healthcare data.
  • Data Privacy Laws – GDPR, CCPA, and internal enterprise policies.

✅ Even Internal Apps Must Be Secure

Sensitive fields like:

  • User name
  • Email address
  • Event details
  • Contact numbers

… must be protected even if the app is only used internally.

Failure to secure them can result in:

  • Regulatory penalties
  • Data breach notifications
  • Loss of customer trust

🔷 Authentication: Identifying Users

Authentication = verifying the identity of the user.

✅ Common Authentication Mechanisms

  1. Username + Password
    • Most common, but easily compromised if not hashed and protected.
  1. Two-Factor Authentication
    • Something you know (password) + something you have (OTP/token).
  1. Biometric / Hardware-Based
    • RSA tokens, fingerprint, face recognition, etc.
  1. certificate..
  1. ….

In the Calendar app:

  • Authenticated users should see their events.
  • Anonymous users should be restricted to public views.

🔷 Authorization: Granting Access

Authorization = deciding what a user is allowed to do.

✅ Role-Based Access Control (RBAC)

  • Users are assigned roles like:
    • ROLE_USER, ROLE_ADMIN, ROLE_MANAGER
  • Access control is applied to:
    • HTTP endpoints (URL paths)
    • Controller methods
    • View logic (e.g., show/hide menu links)
    • Domain logic (e.g., "only owner can edit")

In the app:

  • All users can access everything — there’s no role check.
  • A user can even edit someone else’s calendar event.

🔷 Critical Technical Vulnerabilities in the App Code

  1. No Authentication
    • Users can browse the app as "anonymous."
    • There's no login form, no session-based user context.
  1. No Authorization
    • No role checks, no guards.
    • Admin sections are exposed.
  1. Plaintext Passwords
    • Passwords are stored raw in the database.
    • No hashing, no salt.
  1. Config File Secrets
    • Credentials (like DB password) are hardcoded in .properties files.
  1. No HTTPS
    • App is served over HTTP.
    • Network sniffers can steal session cookies and credentials.
  1. No CSRF/XSS Protections
    • Forms can be posted from anywhere.
    • Inputs are not escaped or sanitized.
  1. No Resource Ownership
    • No logic to restrict users to their own events.
    • User A can view/edit/delete User B’s events by manipulating URLs.

🔷 What Spring Security Will Help You Do

Spring Security helps fix these flaws by providing secure defaults and deep customization.

✅ Features Spring Security Provides

  • Authentication:
    • Form login
    • Basic Auth
    • OAuth2 and OpenID
    • JWT support
  • Authorization:
    • Role-based URL and method protection
    • Access control expressions
  • Session Management:
    • Timeout settings
    • Concurrent session control
    • Session fixation protection
  • CSRF Protection:
    • Enabled by default
    • Tokens in every form
  • XSS Prevention:
    • Output encoding in Spring MVC views
    • Input sanitization with validators
  • Password Management:
    • Secure hashing using BCryptPasswordEncoder
    • Avoids storing raw passwords
  • HTTPS Support:
    • Secure headers
    • SSL enforcement and redirection
