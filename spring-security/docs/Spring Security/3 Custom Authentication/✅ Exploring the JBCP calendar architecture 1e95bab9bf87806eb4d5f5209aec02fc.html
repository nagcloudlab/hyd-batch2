✅ Exploring the JBCP calendar architecture

🔷 Application Context

  • 📦 Previous examples used Spring Framework with manual configuration.
  • ✅ This chapter transitions to Spring Boot for faster setup and less boilerplate.
  • 🔄 But the core Spring Security concepts remain unchanged between Spring Boot and classic Spring Framework.

🧩 1. Domain Model Objects

These objects represent real application data and are separate from Spring Security types. This separation makes the app testable, reusable, and easy to secure without polluting domain logic.

✅ CalendarUser – Serializable Domain User

📄 com.example.calendarapp.domain.CalendarUser



public class CalendarUser implements Serializable {
    private Integer id;
    private String firstName;
    private String lastName;
    private String email;
    private String password;
}

🔍 Key Features:

  • Represents a real business user, not a UserDetails.
  • ✅ Implements Serializable for session storage or distributed caching.
  • Managed and persisted via DAO layer (CalendarUserDao).
  • Doesn’t include roles or authorities — that comes from Spring Security later.

✅ Event – A Calendar Event with Validation

📄 com.example.calendarapp.domain.Event



public record Event(
    Integer id,
    @NotEmpty String summary,
    @NotEmpty String description,
    @NotNull Calendar dateWhen,
    @NotNull CalendarUser owner,
    CalendarUser attendee
) {}

🔍 Key Features:

  • Written using Java record (immutable data holder).
  • Built-in validation annotations (like @NotNull, @NotEmpty) ensure correctness.
  • Every event has:
    • An owner (required)
    • An attendee (optional)
    • A scheduled date (dateWhen)

🛠 2. Service Layer – CalendarService

Acts as a business-facing API over your DAO layer.

📄 com.example.calendarapp.service.CalendarService



public interface CalendarService {
    CalendarUser getUser(int id);
    CalendarUser findUserByEmail(String email);
    List<CalendarUser> findUsersByEmail(String partialEmail);
    int createUser(CalendarUser user);

    Event getEvent(int eventId);
    int createEvent(Event event);
    List<Event> findForUser(int userId);
    List<Event> getEvents();
}

🔍 Key Features:

  • Abstracts DB interactions through the DAO layer.
  • Provides consistent access to user and event logic.
  • Exposes email-based search, user creation, and event management.

👤 3. Managing the Logged-in User – UserContext

✅ UserContext Interface

📄 com.example.calendarapp.security.UserContext



public interface UserContext {
    CalendarUser getCurrentUser();
    void setCurrentUser(CalendarUser user);
}

🧠 Why It’s Smart:

  • Completely decouples domain code from Spring Security internals like Authentication or SecurityContextHolder.
  • Makes unit testing and future migration easier (e.g., to JWT or another auth mechanism).
  • Enables mocking the logged-in user in early development.

❌ UserContextStub – Fixed User for Development

📄 com.example.calendarapp.security.UserContextStub



@Component
public class UserContextStub implements UserContext {
    private final CalendarUser dummyUser = new CalendarUser(...);

🔍 Use Cases:

  • ✅ Allows testing the UI without login logic.
  • 🔁 Replace with a real implementation (SpringSecurityUserContext) when authentication is wired.

✅ SpringSecurityUserContext – Real-Time Security Integration

📄 com.example.calendarapp.security.SpringSecurityUserContext



@Component
public class SpringSecurityUserContext implements UserContext {

    private final CalendarService calendarService;

    public CalendarUser getCurrentUser() {
        Authentication auth = SecurityContextHolder.getContext().getAuthentication();

        if (auth == null || !(auth.getPrincipal() instanceof User user)) {
            return null;
        }

        String email = user.getUsername();
        return calendarService.findUserByEmail(email);
    }

    public void setCurrentUser(CalendarUser user) {
        throw new UnsupportedOperationException("Not supported.");
    }
}

🧠 Design Highlights

ConceptReason
✅ Uses SecurityContextHolderFollows Spring Security best practices
✅ Casts to Spring UserExtracts email from the logged-in user
✅ Converts to domain userUses service lookup to get full CalendarUser
✅ Keeps domain code cleanDoes not leak Spring Security internals

🧠 Best Practices in This Design

  • ⚠️ Avoid using email as a user ID in real apps — prefer UUID or internal ID.
  • ✅ Always null-check the Authentication and principal before casting.
  • ✅ Keep app code working with your own domain objects, not Spring's UserDetails.
