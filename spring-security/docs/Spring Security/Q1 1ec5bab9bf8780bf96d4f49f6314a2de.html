Q1

🎯 Problem Statement

  • A Spring Boot microservice handles high request volume
  • For each request, it must call a third-party API
  • The third-party API limits requests to 100 per second
  • Exceeding the limit results in failures or throttling
  • We need to:
    • Respect the 100/sec limit
    • Queue and retry excess requests

Possible solutions

✅ 1. In-Memory Queue + RateLimiter (What We Built)

How it works:

Queue user requests in memory → Worker + RateLimiter processes queue at 100/sec → Retries on failure.

Tech: Spring Boot + Resilience4j + BlockingQueue + ExecutorService

✅ Pros:

  • Simple to implement
  • No external dependencies
  • Great for small-scale or single-instance services

⚠️ Cons:

  • Queue is not persistent (lost on restart)
  • Doesn’t scale horizontally
  • Memory-bound

🚀 2. Distributed Queue (Kafka/RabbitMQ) + Worker Pool

How it works:

Push user requests to Kafka or RabbitMQ → A dedicated worker service/process pulls at 100/sec rate → Calls third-party.

✅ Pros:

  • Highly scalable
  • Decouples producer (web service) and consumer (worker)
  • Persistent, reliable queueing
  • Allows analytics/retries/reprocessing

⚠️ Cons:

  • Requires setting up Kafka/RabbitMQ
  • Higher complexity
  • Message deduplication and retry handling needed

🧠 3. Async Processing via Webhooks or Callback Queues

How it works:

Accept request → Respond with 202 Accepted → Queue it → Process later → Use webhook/callback when done

✅ Pros:

  • Immediate response to users (non-blocking)
  • Ideal when third-party operation is not time-sensitive

⚠️ Cons:

  • More complex API design
  • Client must support async callbacks
  • Harder to track job state unless you store it

🛑 4. API Gateway with Rate Limiting

How it works:

Use API Gateway (like Kong, NGINX, Spring Cloud Gateway) to rate-limit outbound requests to the third-party API.

✅ Pros:

  • Centralized control
  • Easy to manage throttling and retries
  • Can apply rate limit per route

⚠️ Cons:

  • Can’t queue by itself (needs queueing backend)
  • Typically throttles or rejects, doesn't hold/queue excess

☁️ 5. Use Cloud Task Queues / Managed Queues

How it works:

Use GCP Pub/Sub, AWS SQS, Azure Queue → Trigger workers → Add rate-limiting logic in serverless functions

✅ Pros:

  • Managed infra
  • Resilient and scalable
  • Ideal for cloud-native solutions

⚠️ Cons:

  • Tied to cloud ecosystem
  • Latency and pricing considerations

🧩 6. Token Bucket Algorithm with Redis

How it works:

Use Redis to implement a shared, distributed rate limiter (token bucket) across multiple instances.

✅ Pros:

  • Works across distributed systems
  • Avoids overloading even in multi-instance deployments

⚠️ Cons:

  • Requires Redis and correct locking logic
  • Doesn’t handle queueing; needs to combine with a message queue
