A little bit of polish

🔷 A Little Bit of Polish – Improving the App UX

We made progress with a working login but a few UX issues remain:

  • Login form is basic and unbranded.
  • All users see links like “All Events”, even without permission.
  • The app does not show user identity (e.g., “Welcome user@example.com”).
  • There’s no clear logout button.
  • Hardcoded users aren't sustainable.

This section focuses on customizing these behaviors before moving to advanced security.

✅ Customizing Login Page

🔧 Replace default Spring login page with your own HTML login form.

SecurityConfig.java:


.formLogin(form -> form
    .loginPage("/login/form")
    .loginProcessingUrl("/login")
    .failureUrl("/login/form?error")
    .usernameParameter("username")
    .passwordParameter("password")
)

📌 What each method does:

  • loginPage(): URL that displays the login form.
  • loginProcessingUrl(): URL that receives POSTed credentials.
  • failureUrl(): Redirect after login failure.
  • usernameParameter(), passwordParameter(): Form field names (default: j_username, j_password).

👉 Why customize?

  • Avoid exposing /spring_security_login (the default).
  • Prevent attackers from fingerprinting Spring apps.
  • Improve branding and user experience.

✅ Designing a Custom Login Page (login.html)

Your form must:

  • Submit to /login via POST
  • Use field names username and password
  • Display error/success using:

<div class="alert" th:if="${param.error != null}">
  Failed to login
</div>
<div class="alert" th:if="${param.logout != null}">
  You have been logged out
</div>

📌 Use SPRING_SECURITY_LAST_EXCEPTION to show specific error messages.

✅ Preventing the Infinite Redirect Loop

🚨 Issue:

  • If /login page itself requires authentication, you’ll get stuck in a redirect loop.

🔧 Fix:


.requestMatchers("/login/**").permitAll()

📌 Ensure this is set before applying .authenticated() or .hasRole() globally.

✅ Configuring Logout

Spring Security supports logout with minimal config:


.logout(form -> form
  .logoutUrl("/logout")
  .logoutSuccessUrl("/login?logout")
)

🧠 Default method is POST to /logout.

✅ Show logout success message on login page using:


<div th:if="${param.logout != null}">
  You have been logged out.
</div>

✅ Adding Logout Link to Template


<a th:href="@{/logout}">Logout</a>

🧠 It’s just a link, but Spring handles it securely under the hood.

✅ Role-Based Access Control

Customize who sees what:


.authorizeHttpRequests(authz -> authz
  .requestMatchers("/admin/**").hasRole("ADMIN")
  .requestMatchers("/events/").hasRole("ADMIN")
  .requestMatchers("/**").hasRole("USER")
)

You can also allow static resources to be served without authentication:


.requestMatchers("/resources/**").permitAll()
.requestMatchers("/webjars/**").permitAll()

✅ Post-Login Behavior

Spring Security supports two flows:

  1. If user accessed a protected resource → Redirects them back.
  1. If not, sends to defaultSuccessUrl().

🛠 To control this:


.defaultSuccessUrl("/default")

✅ Redirect After Login Based on Role

Create DefaultController.java:


@RequestMapping("/default")
public String defaultAfterLogin(HttpServletRequest request) {
  if (request.isUserInRole("ADMIN")) {
    return "redirect:/events/";
  }
  return "redirect:/";
}

🧠 This checks the user’s role using HttpServletRequest.isUserInRole(...).

✅ Benefit: Decouples logic from Spring-specific APIs.

✅ Optional: Always Redirect to Default Page


.defaultSuccessUrl("/default", true)

📌 Bypasses Spring’s saved request cache.

✅ Final Testing

Try these flows:

  • Login with valid/invalid credentials.
  • Observe role-based redirection.
  • Ensure logout works.
  • See login failure messages.
  • Test loop avoidance by verifying login page loads anonymously.
