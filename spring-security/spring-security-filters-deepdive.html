Servlet Filters

The Filter Interface

javax.servlet.Filter (Jakarta EE: jakarta.servlet.Filter)

Java
public interface Filter {

    void init(FilterConfig config);         // called once at startup

    void doFilter(
        ServletRequest request,
        ServletResponse response,
        FilterChain chain                     // the chain of remaining filters
    );

    void destroy();                          // called once at shutdown
}
Key method: doFilter(). You can:
1. Inspect/modify the request before passing it on
2. Call chain.doFilter(request, response) to continue the chain
3. Inspect/modify the response after the servlet runs
4. Or block the request entirely (don't call chain.doFilter)
Servlet Filters

A Simple Filter Example

Java
public class LoggingFilter implements Filter {

    @Override
    public void doFilter(ServletRequest request, ServletResponse response,
                          FilterChain chain) throws IOException, ServletException {

        HttpServletRequest req = (HttpServletRequest) request;

        // BEFORE — runs before the servlet
        System.out.println("Incoming: " + req.getMethod() + " " + req.getRequestURI());

        // PASS IT ON — continue to next filter or servlet
        chain.doFilter(request, response);

        // AFTER — runs after the servlet has responded
        System.out.println("Response status: " + ((HttpServletResponse) response).getStatus());
    }
}
chain.doFilter() is the key call. It invokes the next filter in the chain. If you don't call it, the request is blocked — the servlet never executes.
Servlet Filters

The FilterChain

Filters execute in a strict, configured order

Request → Filter A → Filter B → Filter C → Servlet → Filter C → Filter B → Filter A → Response
Nested execution: Each filter wraps around the next. Like Russian dolls. The outermost filter sees the request first and the response last. This is why it's called a chain.
Pseudocode
// Filter A
doFilter(req, res, chain) {
    // A: before
    chain.doFilter(req, res);  // calls Filter B
        // B: before
        chain.doFilter(req, res);  // calls Filter C
            // C: before
            chain.doFilter(req, res);  // calls Servlet
            // C: after
        // B: after
    // A: after
}

web.xml

Order in web.xml = execution order. Filters listed first run first. In Spring Boot, you use @Order or FilterRegistrationBean instead.
Servlet Filters

The Problem

Servlet Filters live in the Servlet Container world. Spring beans live in the Spring world.

Servlet Container World

Tomcat manages Servlet Filters.
Doesn't know about Spring beans.
Can't inject dependencies.
Can't use @Autowired.

Spring ApplicationContext World

Spring manages beans.
Full DI support.
All your services, repos, configs.
But Tomcat doesn't see these beans.

How do we use Spring-managed security logic inside a Servlet Filter? We need a bridge. That bridge is DelegatingFilterProxy.

How DelegatingFilterProxy Works

Step by step — what happens on every request

Simplified internals
public class DelegatingFilterProxy extends GenericFilterBean {

    private String targetBeanName = "springSecurityFilterChain";

    @Override
    public void doFilter(ServletRequest req, ServletResponse res, FilterChain chain) {

        // 1. Look up the Spring bean from ApplicationContext
        Filter delegate = applicationContext.getBean(targetBeanName, Filter.class);

        // 2. Delegate the actual filtering to the Spring bean
        delegate.doFilter(req, res, chain);
    }
}
That's it! DelegatingFilterProxy is just a bridge. It finds a Spring-managed Filter bean by name and delegates. All the real security logic lives in the Spring bean.
Bridge

Spring Boot Auto-Configuration

In Spring Boot, you never register DelegatingFilterProxy manually

Without Spring Boot (manual)

You must create SecurityWebAppInitializer extends AbstractSecurityWebApplicationInitializer. This registers DelegatingFilterProxy as a Servlet Filter with @Order(1).

With Spring Boot (automatic)

SecurityFilterAutoConfiguration registers DelegatingFilterProxy automatically when spring-boot-starter-security is on the classpath. Zero config needed.

Spring Boot magic: You just add the dependency. Everything is wired. But understanding this plumbing helps you debug and customize.

Part 3

FilterChainProxy — The Security Dispatcher

FilterChainProxy

What is FilterChainProxy?

The Spring-managed bean that DelegatingFilterProxy delegates to

Single entry point

All security processing starts here. Registered as the bean named "springSecurityFilterChain".

Holds multiple chains

Contains a List<SecurityFilterChain>. Selects the right chain for each request URL.

URL matching

First SecurityFilterChain whose request matcher matches wins. Order matters!

Debugging single point

Add a breakpoint here to see ALL security processing. One place to debug everything.

FilterChainProxy

Multiple SecurityFilterChains

Different URL patterns can have completely different security configs

FilterChainProxyReceives request from DelegatingFilterProxy
↓ which chain matches?
SecurityFilterChain #1: /api/**Stateless, JWT filter, no CSRF, no sessions
SecurityFilterChain #2: /admin/**Form login, CSRF enabled, session-based
SecurityFilterChain #3: /**Default catch-all chain
Java
@Bean @Order(1)
SecurityFilterChain apiChain(HttpSecurity http) { // matches /api/** }

@Bean @Order(2)
SecurityFilterChain adminChain(HttpSecurity http) { // matches /admin/** }

@Bean @Order(3)
SecurityFilterChain defaultChain(HttpSecurity http) { // matches /** }

Security Filters

Filter #5: CsrfFilter

Protects against Cross-Site Request Forgery

What it does

On every state-changing request (POST, PUT, DELETE), checks that a valid CSRF token is present. Rejects if missing or invalid.

How it works

Server generates a random token per session. Token must be included in form (_csrf hidden field) or header (X-CSRF-TOKEN).

Flow
// CsrfFilter pseudocode
if (request.method is GET/HEAD/OPTIONS) {
    chain.doFilter(req, res);  // skip — safe methods
} else {
    CsrfToken expected = tokenRepository.loadToken(request);
    String actual = request.getHeader("X-CSRF-TOKEN");
    if (!expected.getToken().equals(actual)) {
        throw new InvalidCsrfTokenException();  // 403!
    }
    chain.doFilter(req, res);
}

Security Filters

Filter #13: ExceptionTranslationFilter

Catches security exceptions and translates them to HTTP responses

Pseudocode
try {
    chain.doFilter(req, res);  // calls AuthorizationFilter (next)

} catch (AuthenticationException e) {
    // User is NOT authenticated
    authenticationEntryPoint.commence(req, res, e);
    // Form login: redirect to /login
    // HTTP Basic: 401 + WWW-Authenticate header

} catch (AccessDeniedException e) {
    if (isAnonymous) {
        authenticationEntryPoint.commence(req, res, e);  // 401
    } else {
        accessDeniedHandler.handle(req, res, e);          // 403
    }
}
Key insight: ExceptionTranslationFilter wraps around AuthorizationFilter. If AuthorizationFilter throws, this filter catches and handles it.

Custom Filters — Where JWT Lives

Custom Filters

Adding Your Own Filter

Insert a custom filter at a specific position in the chain

Java
// Insert BEFORE UsernamePasswordAuthenticationFilter
http.addFilterBefore(
    jwtAuthFilter,
    UsernamePasswordAuthenticationFilter.class
);

// Insert AFTER a specific filter
http.addFilterAfter(loggingFilter, CsrfFilter.class);

// Replace a filter at the same position
http.addFilterAt(myAuthFilter, UsernamePasswordAuthenticationFilter.class);
addFilterBefore is the most common. For JWT: your filter runs before Spring tries form login. If JWT is valid, Spring's filter is skipped (auth already set in SecurityContext).

Custom Filters

JWT Filter Implementation

OncePerRequestFilter guarantees one execution per request

Java
public class JwtAuthFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest request,
            HttpServletResponse response, FilterChain chain) {

        String header = request.getHeader("Authorization");

        if (header != null && header.startsWith("Bearer ")) {
            String token = header.substring(7);
            if (jwtUtil.isValid(token)) {
                // Extract user info from token
                String username = jwtUtil.getUsername(token);
                var authorities = jwtUtil.getAuthorities(token);

                // SET authentication — this is the key line!
                var auth = new UsernamePasswordAuthenticationToken(
                    username, null, authorities);
                SecurityContextHolder.getContext().setAuthentication(auth);
            }
        }
        chain.doFilter(request, response);  // continue chain
    }
}
Custom Filters

Why OncePerRequestFilter?

Not javax.servlet.Filter — and for good reason

javax.servlet.Filter

May execute multiple times per request (forwards, includes, error dispatches). You'd need to check manually.

OncePerRequestFilter

Spring guarantees exactly once per request. Handles dispatches internally. Also provides shouldNotFilter() to skip certain URLs.

Java
// Skip the filter for certain URLs
@Override
protected boolean shouldNotFilter(HttpServletRequest request) {
    return request.getRequestURI().startsWith("/public");
}

Key Takeaways

Servlet Filters are the foundation

Spring Security IS a set of Servlet Filters. Understanding Filter & FilterChain is essential.

DelegatingFilterProxy bridges worlds

Connects Servlet Container to Spring ApplicationContext. Auto-configured by Spring Boot.

FilterChainProxy dispatches

Holds multiple SecurityFilterChains. URL pattern matching. First match wins.

~15 filters, strict order

SecurityContextHolder → CSRF → Authentication → Exception → Authorization. Each has ONE job.

Custom filters slot in

addFilterBefore/After/At. JWT filters go before UsernamePasswordAuthFilter.

ExceptionTranslation wraps Authorization

401 for unauthenticated. 403 for unauthorized. The catch block that translates exceptions.
